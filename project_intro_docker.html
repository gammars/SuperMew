🐳 SuperMew 本地运行说明 · Docker / uv / npm

这个项目不是一个单独脚本，而是一套 RAG 聊天应用。

SuperMew 由 Vue 前端、FastAPI 后端、PostgreSQL、Redis、Milvus 向量库和大模型接口组成。 Docker 在这里负责把数据库、缓存和向量库这些“后台基础设施”一次性拉起来。

前端：Vue 3 + Vite 后端：FastAPI + LangChain 依赖：Docker Compose 包管理：uv + npm 向量库：Milvus

一句话看懂运行架构

浏览器

访问 127.0.0.1:8000，使用聊天和文档上传页面。

FastAPI 后端

由 uv 启动，处理登录、聊天、文档上传、SSE 流式输出。

LangChain Agent

决定是否调用天气工具或知识库检索工具。

RAG 知识库

文档切块、embedding、Milvus 检索、rerank、生成回答。

Docker 服务

PostgreSQL、Redis、Milvus、MinIO、Etcd 在后台提供基础能力。

Docker 在这个项目里到底部署了什么？

执行 docker compose up -d 后，Docker 会根据根目录的 docker-compose.yml 创建一个名为 milvus 的网络，并启动下面这些容器。

容器名 镜像 端口 作用 持久化数据目录
supermew-postgres postgres:15 5432 关系数据库，存用户、会话、聊天历史、父级文档块。 volumes/postgres
supermew-redis redis:7-alpine 6379 缓存热点会话和父文档块，加速读取。 volumes/redis
milvus-etcd quay.io/coreos/etcd:v3.5.18 内部使用 Milvus 的元数据组件，类似“向量库的目录账本”。 volumes/etcd
milvus-minio minio/minio 9000, 9001 Milvus 使用的对象存储，保存向量库底层文件。 volumes/minio
milvus-standalone milvusdb/milvus:v2.5.14 19530, 9091 核心向量数据库，保存文档 embedding，负责 Dense + BM25 混合检索。 volumes/milvus
milvus-attu zilliz/attu:v2.5.11 8080 或 18080 Milvus 的网页管理工具。它不是核心服务，起不来也不影响后端聊天。 无关键业务数据

这些东西到底放在哪里？

C

Docker 镜像和容器层：通常在 C 盘

Windows + Docker Desktop + WSL2 下，镜像通常进入 Docker 自己的 WSL 虚拟磁盘， 常见位置类似 C:\Users\PC\AppData\Local\Docker\wsl\。 这就是你看到 C 盘变大的主要原因。

D

项目运行数据：在当前项目目录

这个项目的 compose 文件把数据库持久化目录映射到了 D:\大学内容\大三下\项目分析\SuperMew\volumes。 停掉容器后，这些数据仍然保留。

Py

Python 依赖：项目的 .venv

uv sync --frozen 会创建/使用 .venv， FastAPI、LangChain、pymilvus、sentence-transformers 都在这里。

JS

前端依赖：frontend/node_modules

npm.cmd install 安装 Vue、Vite、Axios 等前端包； npm.cmd run build 会生成 frontend/dist。

如果 C 盘紧张，可以在 Docker Desktop 的 Settings → Resources → Advanced 里调整 Disk image location 到 D 盘。

第一次完整启动命令

1. 启动 Docker 依赖

拉起数据库、缓存、向量库。-d 表示后台运行。

cd D:\大学内容\大三下\项目分析\SuperMew
docker compose up -d

2. 安装后端依赖

按 uv.lock 安装 Python 依赖，避免版本飘。

uv sync --frozen

3. 安装并构建前端

生成 frontend/dist，后端会托管这份静态页面。

cd frontend
npm.cmd install
npm.cmd run build
cd ..

4. 启动后端

启动 FastAPI。看到 Application startup complete 就算成功。

uv run python -m uvicorn backend.app:app --host 127.0.0.1 --port 8000 --log-level debug

日常常用 Docker 命令

查看容器状态

docker compose ps

看 Postgres、Redis、Milvus 是否 running / healthy。

启动后台服务

docker compose up -d

日常开项目时先执行它。

停止服务

docker compose down

停止并删除容器，但保留 volumes 数据。

查看占用空间

docker system df

看镜像、容器、缓存占了多少磁盘。

查看日志

docker compose logs -f standalone

重点排查 Milvus 时用。Postgres/Redis 也可以换服务名看。

清理不用的缓存

docker system prune

会清理没用的缓存。prune -a 更彻底，但下次可能要重新下载镜像。

几个你已经遇到过的坑

Attu 的 8080 端口冲突

如果看到 ports are not available: 8080，把 compose 里的 8080:3000 改成 18080:3000。

HuggingFace 模型下载失败

BAAI/bge-m3 是本地 embedding 模型。首次启动会下载约 2GB 权重；HF_ENDPOINT 要能访问。

CUDA 报错

如果 PyTorch 是 CPU 版，EMBEDDING_DEVICE 要写 cpu，不要写 cuda。

Milvus collection not found

服务已经启动，但还没上传文档。先上传 PDF/Word/Excel，让系统创建 embeddings_collection。

判断项目是否真的跑起来

  • 后端成功：终端出现 Application startup complete。
  • 前端成功：浏览器打开 http://127.0.0.1:8000/ 有页面。
  • 数据库成功：注册/登录接口返回 200。
  • 知识库成功：上传文档后，Milvus 创建 embeddings_collection，聊天不再报 collection not found。
