RAG · Agent · FastAPI · Vue · Docker · Milvus

SuperMew：面向文档知识库的流式智能问答系统

本项目将前端聊天界面、FastAPI 后端、LangChain Agent、Milvus 混合检索、 PostgreSQL 持久化和 Redis 缓存组合成一套可本地运行的 RAG 应用。 它既能进行普通对话，也能基于上传文档完成知识库检索、重写、评分、合成和流式回答。

🧭 RAG 流程可视化 ⚡ SSE 流式输出 📚 文档上传与向量化 🧱 Milvus Hybrid Search 🔐 登录与权限控制 🐳 Docker 本地依赖

项目背景与核心模块

项目目标是构建一个“能看文档、能解释检索过程、能流式回答”的本地知识库助手。 它适合课程项目展示、RAG 技术学习、保研/资料类文档问答、以及多阶段检索流程演示。

前

Vue 前端

提供登录、会话、聊天、文档上传、RAG 过程展示和来源片段查看。生产构建产物位于 frontend/dist。

API

FastAPI 后端

暴露认证、会话、聊天、文档接口；通过 /chat/stream 实现流式回答。

AI

LangChain Agent

负责调用大模型、选择工具，并在需要时调用 search_knowledge_base 进入 RAG。

RAG

检索增强生成

支持复杂度分类、问题分解、混合检索、Auto-merging、相关性评分、查询重写和最终上下文合成。

DB

数据层

PostgreSQL 保存用户、会话、父级文档块；Redis 做缓存；Milvus 保存叶子分块向量。

Ops

运行依赖

Docker Compose 启动 PostgreSQL、Redis、Milvus、Etcd、MinIO、Attu；uv 管 Python 环境，npm 管前端依赖。

请求与响应 Agent 调用 RAG 检索与重写

一次完整请求的生命周期：启动 → 核心处理 → 结果输出

1

启动阶段：把“地基”先跑起来

开发者先启动 Docker Desktop，然后执行 docker compose up -d。 Compose 会启动 PostgreSQL、Redis、Milvus、Etcd、MinIO、Attu。 这些服务不是业务代码本身，而是后端运行时依赖的“基础设施”。 

docker compose up -d
uv run --no-sync python -m uvicorn backend.app:app --host 127.0.0.1 --port 8000 --log-level debug
2

后端初始化：加载配置、模型和路由

FastAPI 读取 .env，初始化数据库表、聊天模型、embedding 模型和路由。 如果 EMBEDDING_DEVICE=cuda，本地 embedding 会尝试使用 GPU；如果 PyTorch 是 CPU 版，就会出现 CUDA 相关错误。

3

文档入库：把资料变成可检索知识

用户上传文档后，后端解析文件并做三级分块。父级块写入 PostgreSQL，叶子块生成向量后写入 Milvus。 Milvus 还会通过 BM25 函数支持稀疏检索，最终形成 Dense + Sparse 的混合检索基础。

4

聊天请求：前端通过 SSE 等待流式回答

用户提问时，前端调用 /chat/stream。后端先校验用户身份，读取会话历史， 然后交给 Agent 判断是否需要调用知识库工具。

5

RAG 核心处理：检索、合并、评分、重写

RAG 会先判断问题复杂度。简单问题直接检索；复杂问题会被拆成多个子问题并行检索。 每次检索会经历 Milvus 召回、Auto-merging 合并父块、可选 rerank 精排、文档相关性评分。 如果相关性不足，会触发 step-back 或 HyDE 等重写策略。

6

结果输出：生成回答并展示检索过程

后端将 top-k 文档片段拼接为上下文，调用大模型生成答案，并通过 SSE 逐 token 推送给前端。 同时，检索 trace 会展示候选池数量、是否 auto-merge、是否 rerank、评分结果和来源片段。

关键瓶颈与常见异常

这套系统的瓶颈一般不在单个地方，而是在“本地模型、向量库、外部模型 API、结构化输出兼容性”之间切换。

现象 常见原因 处理方式
Docker 服务起不来 Docker Desktop 未运行、WSL2 后端未启动、端口冲突，如 Attu 占用 8080。 先启动 Docker Desktop；用 docker compose ps 查看状态；8080 冲突可改成 18080:3000。
Milvus collection not found 还没有上传文档，embeddings_collection 尚未创建。 先上传一份 PDF / Word / Excel，等待向量化入库完成。
Torch not compiled with CUDA enabled 项目环境里的 PyTorch 是 CPU 版，但 EMBEDDING_DEVICE=cuda。 安装 CUDA 版 torch，并用 uv run --no-sync 启动，避免 uv 自动同步回 CPU 版。
HuggingFace 模型下载失败 BAAI/bge-m3 首次启动需要下载，网络、镜像源或缓存目录权限异常。 检查 HF_ENDPOINT，可设置 HF_HOME 到项目 D 盘目录。
structured-output fallback 部分 OpenAI-compatible 模型不完全支持 LangChain structured output。 系统已回退为普通文本解析，例如从 yes、complex、【简单问题】 中提取控制信号。
Rerank 404 RERANK_BINDING_HOST 写成了带 /v1 的地址，导致最终变成 /v1/v1/rerank；或者服务商根本不支持 rerank。 填写主机根地址，如 https://api.siliconflow.cn；没有 rerank 服务时留空关闭。

常见问题问答

1. Docker 在这个项目里到底部署了什么？

Docker Compose 部署 PostgreSQL、Redis、Milvus、Etcd、MinIO 和 Attu。 其中 PostgreSQL 存业务数据，Redis 做缓存，Milvus 存向量，Etcd 和 MinIO 是 Milvus 运行依赖，Attu 是 Milvus 的可视化管理页面。

2. uv、npm、Docker 分别负责什么？

uv 管 Python 后端环境和依赖，npm 管 Vue 前端依赖和构建， Docker 管数据库、缓存和向量库这些后台服务。

3. 为什么复杂问题会出现 synthesis 阶段？

复杂问题会先分解成多个子问题，每个子问题独立检索。最后系统进入 synthesis 阶段，把多个子 Agent 的结果去重、合并，再作为最终上下文。 因此 synthesis 展示的是合成层，不一定直接展示每个子检索的 rerank 细节。

4. Rerank 要怎么配置？

项目代码默认请求 /v1/rerank。如果使用 SiliconFlow 或 Jina 这类兼容接口，建议填写主机根地址： 

RERANK_MODEL=BAAI/bge-reranker-v2-m3
RERANK_BINDING_HOST=https://api.siliconflow.cn
RERANK_API_KEY=你的_rerank_api_key
RERANK_MIN_SCORE=0.0

如果没有真实 rerank 服务，建议把三个 rerank 配置留空，系统会退回 Milvus Hybrid + RRF 排序。

5. 为什么使用 GPU 后要加 --no-sync？

当前 uv.lock 可能锁定的是 CPU 版 torch。手动安装 CUDA 版 torch 后，如果继续普通 uv run， uv 可能自动同步环境并把 torch 改回 CPU 版。使用 uv run --no-sync 可以避免启动前自动 sync。

6. 检索结果中的 Auto-merging 是什么？

系统优先召回 L3 叶子块。当同一个父块下命中多个叶子块时，系统会把它们合并回 L2 或 L1 父块， 让上下文更完整，减少答案只看到碎片信息的问题。

展示建议： 如果用于技术分享，可以先讲 Docker/uv/npm 分工，再讲文档入库，最后用“用户提问 → Agent → RAG → Milvus → LLM → 流式输出”的链路解释系统协作。
